## Deploying to kind cluster

### Quick rebuild (most common)
```bash
cd /home/jsalomon/src/pc-asset-hub
./scripts/kind-deploy.sh rebuild "kubectl --context kind-assethub"
```

### Script usage
```bash
./scripts/kind-deploy.sh [command] [kube-cmd]
```

- **Commands**: `deploy` | `up` | `rebuild` | `teardown`
- **kube-cmd**: defaults to `oc`. For kind cluster use `"kubectl --context kind-assethub"`

### Key facts
- Kind cluster name: `assethub`
- kubectl context: `kind-assethub`
- Namespace: `assethub`
- Container engine: podman (auto-detected, sets `KIND_EXPERIMENTAL_PROVIDER=podman`)
- API: http://localhost:30080 (NodePort 30080)
- UI: http://localhost:30000 (NodePort 30000)
- Images: `assethub/api-server:latest`, `assethub/ui:latest`, `assethub/operator:latest`
- Image pull policy: `Never` (loaded via `kind load`)
- Operator-driven: operator creates API + UI deployments from AssetHub CR
- The default kubectl context is NOT the kind cluster. Always specify `--context kind-assethub`.

### Manual restart (if script fails at restart step)
```bash
kubectl --context kind-assethub -n assethub rollout restart deployment/assethub-api deployment/assethub-ui deployment/assethub-operator
kubectl --context kind-assethub -n assethub rollout status deployment/assethub-api --timeout=120s
```

### Verify deployment
```bash
kubectl --context kind-assethub -n assethub get pods
curl -s http://localhost:30080/healthz
curl -s http://localhost:30080/readyz
```
