## Problem

When a Go API update handler uses `string` in the request DTO, omitting the field in the JSON body maps to `""` (Go zero value). If the service layer then does `e.Field = req.Field`, it silently erases the existing value. The caller intended "don't change this field" but the backend interpreted it as "set to empty string".

## Example

```go
// BAD: string field — omitting "description" in JSON sets it to ""
type UpdateEnumRequest struct {
    Name        string `json:"name"`
    Description string `json:"description"`  // omitted → ""
}

// Service unconditionally overwrites:
e.Description = description  // erases existing value!
```

## Fix

Use `*string` (pointer) for optional update fields. `nil` means "omitted/no change", `""` means "explicitly set to empty":

```go
// GOOD: pointer field — omitting "description" in JSON sets it to nil
type UpdateEnumRequest struct {
    Name        string  `json:"name"`
    Description *string `json:"description"`  // omitted → nil
}

// Handler preserves current value when nil:
if req.Description != nil {
    desc = *req.Description
} else {
    desc = current.Description  // preserve existing
}
```

## Scope

This bug was found on `UpdateEnum` and fixed. TD-62 tracks auditing ALL update endpoints for the same pattern: `UpdateEntityType`, `UpdateInstance`, `SetParent`, and any future PUT/PATCH handlers.
