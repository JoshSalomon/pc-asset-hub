## The Problem

The `uncovered-new-lines.sh` script has reported 0 uncovered lines when there were actually uncovered new lines. This happened multiple times:

1. Script compared vs HEAD instead of the specified base ref (positional arg parsed as coverage file)
2. Script missed new lines in GORM repo files because git diff line numbers didn't align with coverage block boundaries

## The Fix: Arithmetic Verification

After every coverage run, manually compute uncovered deltas per package:

```
previous_uncovered = previous_total - previous_covered  (from docs/coverage-report.md)
current_uncovered  = current_total  - current_covered   (from go-coverage-table.sh)
delta = current_uncovered - previous_uncovered
```

If `delta > 0`, there ARE uncovered new lines — investigate per-file with `go-uncovered-lines.sh`.
If scripts say 0 but arithmetic says > 0, **THE SCRIPTS ARE WRONG** — investigate manually.

## Example (caught in Phase 2b / US-53)

- `service/meta`: Phase 2 had 780/784 (4 uncov). US-53 showed 801/809 (8 uncov). Delta: +4 new uncovered.
- Script reported 0. Arithmetic caught it. Four error paths in UpdatePin/AddPin were uncovered.
- Fixed by adding error path tests. Final: 805/809 (4 uncov) — back to baseline.

## Rule

Trust math, not scripts. Do the arithmetic check on EVERY coverage run.
