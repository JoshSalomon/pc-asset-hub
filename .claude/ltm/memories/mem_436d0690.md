## 4-Tier Testing Architecture

### Tier 1: Go Backend (`make test-backend`)
- Service tests: mock repos from `internal/domain/repository/mocks/`
- Handler tests: mock services + httptest + Echo
- Repo integration tests: in-memory SQLite via `testutil.NewTestDB(t)`
- Error branches: `closedDB(t)` helper triggers `result.Error` paths
- Operator: `fake.NewClientBuilder()` from controller-runtime
- Coverage: `go test ./internal/... -count=1 -coverprofile=coverage.out` (NO `-coverpkg`)

### Tier 2: UI Browser Tests (`make test-browser`)
- Config: `vitest.browser.config.ts`, real Chromium via Playwright
- Uses `vitest-browser-react` for render, `page` from `vitest/browser` for locators
- `api/client.ts` tested via `vi.stubGlobal('fetch', mockFetch)` — no module mock
- Coverage: `npx vitest run --config vitest.browser.config.ts --coverage`

### Tier 3: System Tests (`make test-system`)
- Config: `vitest.system.config.ts`, runs in Node, requires live kind cluster
- Launches headless Chromium, navigates to `localhost:30000`
- API setup/teardown via Node `fetch()`, unique names with `Date.now()`

### Tier 4: Live Tests (`make test-live`)
- Bash scripts in `scripts/test-*.sh`, run against `localhost:30080`
- 8 scripts covering containment, data viewer, validation, publishing, copy/replace, descriptions, system attributes

### Key Pitfalls
- PatternFly 6 + jsdom: interactive tests timeout → use Browser Mode
- PatternFly Select: use `menuitem` role in tests (not `option`)
- PatternFly Tabs renders ALL tab content hidden via CSS → use `.first()` for duplicates
- Modal buttons: `page.getByRole('dialog').getByRole('button', ...)`
- `getByText()` strict mode: fails on substring matches → use `getByRole` with `{ exact: true }`
- PatternFly `TextInput type="number"` renders as `spinbutton` role, not `textbox`
- Go cross-package coverage: do NOT use `-coverpkg` — causes false uncovered reports
- Coverage files persist between runs: always `rm -f coverage.out` before measuring
