## Problem
Drag-and-drop from Nemo file manager to Chromium-based browsers (Chrome Canary 149+, Brave) fails on Linux X11. `dataTransfer.files` is empty, `dataTransfer.types` contains only `text/plain` with the file path string.

## Root Cause
Nemo sends file drags as `text/plain` file paths via X11 DnD protocol instead of proper file objects. Chrome's Ozone X11 backend does not translate this to HTML5 `Files` type. Firefox handles the same drag correctly.

## What Was Tried (and didn't fix it)
1. Document-level dragenter/dragover preventDefault — events reach the element but types=text/plain
2. Native addEventListener instead of React synthetic events — same result
3. `<input type="file">` overlay with opacity:0 — Chrome still receives text/plain

## Confirmed Working File Managers
- Nautilus (GNOME Files): works
- Thunar (XFCE): works
- Nemo (Cinnamon): BROKEN on Chromium, works on Firefox

## Solution
Graceful fallback: detect `text/plain` without `Files` type in the drop handler, show error message, and auto-open the file picker. Click-to-upload works on all browsers. TD-136 tracks this.

## Key Diagnostic
The `dataTransfer` on drop shows: `files=0, types=text/plain, text/plain="/home/user/path/to/file.json\r\n", items=[{kind:string, type:text/plain}]` — no file objects at all.
