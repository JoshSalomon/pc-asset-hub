## Non-Negotiable Coverage Rule

Josh NEVER approves skipping a testable/coverable line. The ONLY exception he has ever made is one-time database migration code (InitDB column/table drops that only run on first migration from old schema).

### What This Means in Practice

- Do NOT present uncovered lines and ask "fix, defer, or skip?" — just fix them ALL.
- Do NOT categorize lines as "low risk" or "pre-existing pattern" to rationalize skipping.
- Do NOT summarize uncovered lines by category ("24 instance service lines") — list each one individually.
- Do NOT defer UI coverage "to the host session" — UI is part of the application, cover it NOW.
- Do NOT present to the human before attempting every coverable line.
- Do NOT skip the coverage-review skill (Step 8 of coverage-test).

### Incidents That Triggered This Rule

1. Agent identified 43 uncovered backend lines and asked Josh to choose what to fix. Josh: "obviously you need to cover all coverable lines — I am VERY STRICT."
2. Agent skipped UI coverage entirely (179 uncovered lines in new pages) and documented them for "the host session." Josh: "YOU ARE LAZY AND TOTALLY DRUNK... YOUR JOB IS TO WORK AND DO WHAT I ASK YOU."
3. Agent found 3 coverable lines during coverage-review and asked Josh whether to cover them. Josh: "do you need to ask me about 3 lines? have I EVER approved to skip a coverable line?"

### Rule for Skills

The coverage-test and coverage-review skills should check this LTM before completing. The answer to "should I cover this line?" is ALWAYS YES unless it is literally impossible to reach (e.g., useParams guard on a parameterized route, switch default with all cases covered, or one-time migration code explicitly excluded by Josh).
