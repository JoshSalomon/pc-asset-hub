## Problem

During the Import/Export feature (US-55, US-56), approximately 10 bugs were found by the user during manual testing — NOT by automated tests. The bugs spanned export, import, and UI layers. The automated tests were superficial: they tested individual functions in isolation using mocks, but never tested the actual end-to-end flow through JSON serialization.

## Bugs Found by User (Not Tests)

1. **Children not imported** — `ExportInstance.Children` had `json:"-"`, so `json.Unmarshal` silently dropped contained instances. `ParseExportInstances` existed but was never wired into the import flow. No round-trip test existed.

2. **Associations not created for new entity types** — The association creation loop re-queried the DB and found entity types it had just created in the same transaction, incorrectly skipping them. Mock-based tests couldn't reproduce in-transaction visibility.

3. **Bidirectional links from reverse side silently failed** — `CreateAssociationLink` only looked up associations from the source ETV. Bidirectional associations defined on the other entity type were invisible. No test created links from the reverse side.

4. **Export dropped links after ETV versioning** — `assocByID` was built from pinned ETV associations only. After entity type versioning (attribute added), new ETVs got new association IDs via `BulkCopyToVersion`. Links referencing old IDs were silently dropped. Required debug logging in production to diagnose.

5. **Integer attributes lost during import** — Import only set `ValueString`, not `ValueNumber`. Export required `ValueNumber != nil` to include integer attributes. Round-trip dropped all integer values.

6. **Integer attributes exported as strings** — Export preferred `ValueString` over `ValueNumber`, producing `"128000"` instead of `128000` in JSON.

7. **CV label not validated** — UI accepted empty or non-alphanumeric CV labels that the backend rejected.

8. **Catalog/CV conflict not blocking Continue** — The collision step's Continue button explicitly excluded catalog/catalog_version conflicts from its disabled check.

9. **Non-catalog JSON accepted silently** — Uploading any valid JSON (e.g., package.json) showed an empty form with no error.

10. **Confirm message misleading** — "0 identical (will be reused)" shown when nothing was being reused.

## Root Cause

The test strategy had a critical gap: **no round-trip/seam tests**. Unit tests tested each layer independently (export serialization, import logic, JSON parsing) using mocks that couldn't simulate real DB behavior (in-transaction visibility, ETV versioning). The export and import were never tested together through the actual JSON wire format.

## Fix

1. Added `UnmarshalJSON` to `ExportData` for proper children parsing
2. Used `createdETNames` set instead of DB re-query for association creation
3. Added reverse-side bidirectional link lookup in both `CreateAssociationLink` and `createLinksRecursive`
4. Added `GetByID` fallback in export for versioned association IDs
5. Set `ValueNumber` for integer/number attributes during import
6. Export prefers `ValueNumber` over `ValueString` for numeric types
7. Created comprehensive E2E test (`test-import-export-e2e.sh`) that builds realistic schema with versioning, creates all link types, exports, imports with prefix, and verifies every piece matches
8. Updated feat-plan Phase 3 to require round-trip/seam tests for every serialization boundary
9. Updated feat-plan Phase 8 to require `make test-live` before PR creation

## Key Lesson

**Mock-based unit tests cannot catch serialization seam bugs.** When data crosses a boundary (JSON marshal/unmarshal, API wire format, DB transaction), you MUST test the round-trip with the actual serialization, not with mocks that bypass it. The `json:"-"` tag on Children was invisible to any test that constructed `ExportData` directly in Go.
