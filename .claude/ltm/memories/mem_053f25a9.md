## Database Schema (Hybrid fixed + EAV)

### Meta Tables (fixed relational)
- `entity_types` — identity (id, name unique, timestamps)
- `entity_type_versions` — immutable snapshots, UNIQUE(entity_type_id, version)
- `attributes` — belong to entity_type_version (copy-on-write versioning), type: string/number/enum
- `associations` — belong to entity_type_version (NOT entity_type). Types: containment, directional, bidirectional
- `enums` / `enum_values` — first-class reusable enum definitions
- `catalog_versions` — lifecycle_stage: development/testing/production
- `catalog_version_pins` — pins entity_type_version to catalog_version
- `lifecycle_transitions` — audit trail of stage transitions

### Data Tables (EAV pattern)
- `entity_instances` — has parent_instance_id for containment hierarchy, soft deletes, auto-versioning
- `instance_attribute_values` — includes instance_version for full version history, typed columns (value_string, value_number, value_enum)
- `association_links` — links between instances via defined associations

### Critical Design Decisions
1. **Associations are versioned** (tied to entity_type_versions, not entity_types). Adding/removing associations only affects the new version. Old catalog versions see old associations.
2. **Copy-on-write**: Mutating an entity type creates a new version row; all attributes AND associations are copied to the new version.
3. **Name uniqueness**: UNIQUE(entity_type_id, catalog_version_id, parent_instance_id, name) — scoped to parent for contained entities.
4. **Inheritance feasibility**: Can be added later with a nullable parent_entity_type_version_id FK. Additive change, no rearchitecture needed.
