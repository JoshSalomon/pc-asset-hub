## 🔴 Critical / ⚠️ Important

| ID | Item | Current Behavior | Required Behavior |
|----|------|-----------------|-------------------|
| TD-1 | 🔴 Enum deletion safety | Enum delete checks if any attribute references it across all entity type versions (flat check) | Enum cannot be deleted if it is used by any attribute in a **used entity version**. A used entity version is defined as: (1) any entity type version pinned by a catalog version, or (2) the latest version of any entity type (which belongs to an implicit pre-production catalog). Unused historical versions that are not pinned by any CV and are not the latest version should not block deletion. |
| TD-10 | ⚠️ Mutable CVs in development mode | **PARTIALLY RESOLVED.** CV pins are now mutable via US-52 (add/remove/change version). Stage guards added via TD-69: production = blocked, testing = SuperAdmin only, development = RW+. Remaining: pin changes on demoted CVs should trigger re-validation of pinned catalogs. |
| TD-99 | 🐛 Containment cardinality `1..n` not enforced during catalog validation | A container entity type with a containment association of source cardinality `1` or `1..n` to a child type can pass validation even when no contained instances exist. For example, if MCP-Server contains Tools with cardinality `1..n`, creating an MCP-Server instance with zero Tools should fail validation — but it passes. The validation service checks mandatory associations for reference links but does not check containment parent-child counts against the source cardinality constraint. | Extend `CatalogValidationService` to count contained instances (children) per parent instance for each containment association. If the containment cardinality minimum is >= 1 (e.g., `1`, `1..n`) and the parent has zero children of the contained type, report a validation error. |
| TD-102 | ⚠️ Instance table filter has no exact match option — enum values that are substrings of each other can't be filtered precisely | All attribute filters use case-insensitive substring match (LIKE). When enum values are substrings of each other (e.g., "active"/"inactive", "enabled"/"disabled"), filtering by "active" also returns "inactive" instances. Users cannot isolate only "active" items. | Add an exact match toggle to the filter UI (checkbox or mode switch). When enabled, the API passes an `exact=true` parameter and the repo uses `LOWER(value_string) = ?` instead of `LIKE`. Default remains substring (current behavior). Requires: API query param, repo filter mode, UI checkbox per filter column. |
| TD-114 | ⚠️ Changing CV pin version silently orphans instance attribute values and association links | When a CV entity type pin is changed from V1 to V2, copy-on-write gives V2's attributes new IDs. Existing instance attribute values (IAVs) reference V1's attribute IDs and no longer match V2's attributes — values appear lost. The data is still in the database but is unreachable. The same issue affects association links if associations were copied with new IDs. This makes schema evolution (adding/removing attributes) dangerous for catalogs with existing data. | Add an instance attribute migration step when a CV pin version changes. The migration should: (1) **Match attributes by name** between old and new entity type versions. (2) **Remap IAVs** — update `attribute_id` from old to new for matching attributes. (3) **Handle deleted attributes** — IAVs for attributes removed in the new version should be flagged with a warning (not silently dropped). (4) **Handle type changes** — if a matching attribute changed its type definition (e.g., string → integer), warn that existing values may not be valid under the new type. (5) **Handle new required attributes** — warn that existing instances are missing values for newly added required attributes. (6) **Remap association links** similarly if association IDs changed. The migration should run automatically on pin version change with warnings returned to the caller. Alternatively, provide a dry-run mode that shows what would change before committing. (7) **Handle renamed attributes** — if an attribute name changed between versions (same ordinal position, different name), attempt to match by ordinal and warn about the rename so the user can confirm the mapping is correct. Without this, a renamed attribute looks like a delete + add, orphaning the old values. |
