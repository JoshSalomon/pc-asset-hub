Containment — shared lifecycle

Tools live inside their server. Delete the server and all its tools are deleted with it. The server owns the tools it implements.

Reference — independent lifecycle

A Virtual Server picks tools from any server in the catalog. It doesn't own them — it just exposes a curated subset as a single endpoint for a specific use case.

UX Decision Point

Should a Virtual Server reference only tools, or also its source servers? The data model supports indirect resolution (tool → server), but a direct reference may improve the UX. To be validated.

Why it matters

These entity types and relationships aren't hardcoded. They're defined at runtime through the schema manager — so the same system can catalog MCP tools today, and models or prompts tomorrow.

Lifecycle

From draft to production

Catalogs are validated against their schema before publishing. Published catalogs are write-protected.

Draft → Validate → Testing → Production → Publish
Safe Update Path
Copy → Edit → Validate → Testing → Atomic Replace

Data version increments • Old catalog archived for rollback

Why atomic? A catalog is a single versioned entity. You cannot add or change entities individually — the entire catalog must be validated and promoted together.
Gateway Integration — Option 1

Publish once. Gateway watches.

Publishing a catalog creates a Kubernetes Custom Resource. The MCP Gateway controller watches for changes and auto-discovers new catalogs.

Subsequent Update
Catalog updated → dataVersion: 2 → Cache invalidated → Re-fetches metadata

Zero-downtime catalog updates. The Gateway watches the CR — Asset Hub never pushes to the Gateway.

Gateway Integration — Option 2

Plugins that speak the consumer’s language

The mcp-gateway export plugin generates the Gateway’s native CRDs directly. The Gateway never needs to know Asset Hub exists.

Integration Model
Asset Hub → Catalog CR ← Gateway → Fetch data

Gateway must integrate with Asset Hub API to discover and fetch catalog data. Tight coupling between systems.

Export Plugin Model
Asset Hub → mcp-gateway exporter → Native CRDs
MCPServerRegistration MCPVirtualServer

Asset Hub generates the Gateway’s own CRDs. The Gateway sees its native resources — no custom integration, no API calls back to Asset Hub.

Zero coupling. The Gateway treats exported CRDs like any other resource it manages. Asset Hub is the authoring system — the Gateway is just a consumer of standard Kubernetes resources.
Positioning

Asset Hub vs. Backstage Software Catalog

Backstage Asset Hub
Source of truth YAML files scattered across Git repos Centralized database with REST API
Entity model Fixed kinds — Component, API, Resource, System, Domain Dynamic — define any entity type at runtime
Relationships Predefined set (providesApi, dependsOn, partOf, …) Custom associations with types and cardinality
Schema evolution Code changes + deployment for new entity kinds Runtime configuration — no code deployment
Validation YAML schema checks per entity file Cross-entity validation — required attrs, associations, containment
Lifecycle External — managed through Git workflows Built-in — draft → validate → publish with RBAC
The fundamental difference is the source of truth. When Git is your source of truth, schema lives in code, changes go through PRs, and consistency depends on convention. When a purpose-built database is your source of truth, the system itself enforces schema, validates relationships, and manages lifecycle — across all entities, atomically.
